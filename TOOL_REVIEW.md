# Tool review appendix (code + methodology audit)

## Lens: Python correctness review: Roblox API field handling, ISO date parsing, and all metric math in roblox_research.py

The core analytical engine is correct. I recomputed every metric (visits_per_day, ccu_per_1k_visits, _hhi, leader_share_pct, top3_share_pct, winners_1k, demand_ccu, and the four score components) independently from the raw `games` arrays in all 10 E:/roblox-data/data/niche_*.json files and got an EXACT match to the stored `analysis` blocks in every file (e.g. niche_rng demand=787722, ls%=26.4, hhi=1415, w=27; niche_unboxing hhi=7648). Division-by-zero guards (`if demand`, `if total`, `if g.age_days > 0 and g.visits`, `if g.visits`) all behave correctly, and dedup (omni_search `seen` set; `dict.fromkeys` in details/votes/icons; `setdefault` in enrich) is sound. The place->universe resolution, omni-search nextPageToken pagination, batch chunking by DETAILS_CHUNK=50, and the details/votes/discovery merge-fallback chain in _build_game are all logically correct with proper `or`/`.get(default)` fallbacks. The bugs that exist are peripheral: a dead-code block, a few unguarded dict subscripts that could KeyError on malformed API rows, an interpreter-version-dependent date-parse degradation, and an unverifiable genre field-name assumption. None of these corrupt the metric math or the verdict logic that drives the niche recommendations. IMPORTANT for the user's taste anchor: the code is correct, but the headline leader_share/HHI is purely CCU-share arithmetic and faithfully propagates the keyword-contamination bleed-through — e.g. niche_unboxing reports leader='[UPD] +1 Speed Keyboard Escape | Candy & Chocolate' at 87.4% / HHI 7648, which the code would label MONOLITH. That is a DATA artifact, not a code bug; trust per-genre_l2 genre_breakdown and individual game rows over the headline, exactly as the caveat warns.

### Findings
- **[medium] Date parsing of ISO timestamps (_parse_dt, lines 259-265) on Python <= 3.10** — Real Roblox `updated` timestamps carry 7-digit fractional seconds, e.g. niche_rng.json game 'Blox Fruits' has updated='2026-06-15T05:41:13.8540241Z'. datetime.fromisoformat only accepts 0/3/6 fractional digits before Python 3.11. On 3.11+ (this env is 3.14) it parses fine — I verified all three real shapes parse correctly here. On <=3.10 the 7-digit case raises ValueError, which is caught and returns None, so age_days/days_since_update silently fall back to 0. That would zero out is_fresh()/is_stale() and the freshness_pts/staleness_pts score components for affected games without any error — a silent metric degradation depending on interpreter version.
  - _Fix:_ Normalize fractional seconds to 6 digits before parsing (regex-trim the subsecond group) or use a tolerant parser, so behavior is identical across Python versions. At minimum document the 3.11+ requirement, since the failure is silent (score components quietly drop to 0 rather than erroring).
- **[low] cmd_trending _sort tagging (lines 596-598)** — `for g in games: g_dict = asdict(g); g_dict['_sort'] = sid` builds a throwaway dict each iteration, sets a key on it, then discards it. The Game objects are never tagged, and write_json at line 605 re-serializes the untagged objects, so the intended per-feed `_sort` attribution is lost in both the 'all' multi-sort CSV and JSON output. Dead code with a lost-feature side effect (you cannot tell which feed a game came from in the combined output).
  - _Fix:_ Either delete the dead loop, or add a real `_sort`/source field to the Game dataclass (or build the dicts once and write those) so the feed origin survives into the CSV/JSON.
- **[low] Unguarded dict subscripts in batch parsers (details line 188 `out[g['id']]`, votes line 200 `out[v['id']]`, icons line 216 `out[d['targetId']]`)** — These index API rows with hard `[...]` keys instead of `.get(...)`. The module's stated design is that a malformed/partial id is 'skipped, never fatal', but a details/votes row missing `id` (or an icon row missing `targetId`) would raise KeyError and abort the whole chunk/run rather than skipping that one row. Low likelihood in practice (Roblox returns id consistently), but it violates the file's own never-fatal invariant.
  - _Fix:_ Guard with `gid = g.get('id');  if gid is not None: out[gid] = g` (and the analogous form for votes/icons) so one bad row cannot kill the batch.
- **[low] genre_l1 / genre_l2 field-name assumption in _build_game (lines 300-301)** — The merge reads det.get('genre_l1') and det.get('genre_l2'). The classic games.roblox.com/v1/games details payload exposes a single `genre` string (and the newer taxonomy historically used camelCase like genreL1/genreL2 or a nested subgenres object), not snake_case genre_l1/genre_l2. I cannot hit the live API to confirm the current key names, but the saved data has these populated correctly ('RPG'/'Action RPG'), so either the API now returns these exact snake_case keys or the producer post-processed them. If the real key names differ, both fields would silently default to '' and genre_breakdown would collapse every game into '(unknown)' — which is NOT what the saved files show, so this is a verification caveat, not a confirmed bug. Note the genre values themselves are never used in score math, only in genre_breakdown grouping and display.
  - _Fix:_ Confirm the exact detail-endpoint key names against a live response; if they are camelCase/nested, read both spellings (e.g. det.get('genre_l2') or (det.get('subgenres') or [..]) ) so a key rename cannot silently empty the sub-genre breakdown.
- **[low] omni_search pagination edge (lines 151-169)** — Pagination is correct in the normal case (sends nextPageToken back as `pageToken`, stops when token is falsy or pages exhausted, dedups by universeId). One minor edge: if pages==1 the loop runs once, processes the first page, then `token = data.get('nextPageToken')` is fetched and immediately discarded by the `if not token: break` / loop-exit — harmless, no extra request is made. No off-by-one in the page count (range(max(1,pages)) yields exactly `pages` iterations and the post-loop break prevents an extra fetch). This is CORRECT; flagged only to confirm I checked the token round-trip and the param-name mapping (nextPageToken -> pageToken) which is the kind of mismatch that silently returns page 1 forever — it does not here.
  - _Fix:_ No change required. (Confirmed correct.)

## Lens: Research methodology critic — soundness of the opportunity-score / verdict bands as buy/avoid signals for a Roblox niche, grounded in roblox_research.py (analyze_niche, _hhi, genre_breakdown, scoring + verdict thresholds) and the data/niche_*.json files most relevant to the user's RNG-collection and cooking/restaurant taste.

The toolkit is a solid live-demand scanner but its headline opportunity score is a weak buy/avoid signal because three of its four components saturate and stop discriminating, and its concentration math (leader_share/HHI) is computed on a keyword-search cluster that mixes unrelated genres — exactly the contamination the brief warns about. Net effect: almost every niche the user cares about scores 76-86/100 and reads "OPEN & PROVEN," collapsing real differences. Evidence from the JSON: demand_pts hits its 35.0 cap at only ~100k CCU, so niche_food (167k), niche_cooking (110k), niche_rng (788k) and niche_incremental (615k) all get the SAME 35.0 demand score despite an ~8x demand spread. Freshness caps at 2 fresh winners (10 pts each, max 20), so niche_cooking (2 fresh) and niche_food (7 fresh) score an identical 20.0 freshness. That leaves only "openness" doing real work, and openness is built on the contaminated niche-level leader_share/HHI — the single most distorted number in the file. The verdict bands (DEAD / UNPROVEN / MONOLITH / OPEN&PROVEN / OPEN-BUT-COOLING) are directionally reasonable, but the MONOLITH gate (leader_share>=0.6 OR hhi>=5000) runs on the same contaminated cluster, so it both MISSES real per-genre monoliths and is artificially softened by bleed-through games padding the denominator. For the user's actual goal — a lane that sustains 1k+ CCU and is NOT owned by one monolith — the per-genre genre_breakdown and individual games are far more trustworthy than the headline, and they tell a clearer story than the scores: cooking/restaurant Tycoon is a genuinely open, mid-demand lane (Run a Restaurant! only ~19k, genre-level leader_share ~20-34%, 19-21 winners), while "incremental" is more concentrated than its 75.9 score implies once the genre is isolated.

### Findings
- **[high] Concentration math computed on contaminated keyword cluster (analyze_niche leader_share / _hhi)** — leader_share, top3_share and hhi are computed over ALL games returned by one omni_search keyword query, which mixes genres and includes bleed-through mega-titles. In niche_incremental.json the 'leader' is '[UPD] +1 Speed Keyboard Escape | Candy & Chocolate' at 247,976 CCU = 40.3% niche share, hhi 2271 — but that game also surfaces in rng/unboxing/steal-brainrot searches, i.e. it is keyword bleed-through, not the true structural leader of a coherent niche. Conversely, contamination DILUTES concentration: niche_food.json reports leader_share 11.6% / hhi 585 ('fragmented') only because the cluster is padded with off-topic winners like 'Hide or Die!' (14,012), 'My Movie' (12,437), 'Squid Game X' (11,453), 'Feather Family' (6,992) and 'Dig to Escape' (5,103) that have nothing to do with food. The denominator is junk, so both share and HHI are junk. Because 'openness' (the only non-saturated score component) is derived directly from max(leader_share, hhi/10000), the headline score inherits this contamination one-to-one.
  - _Fix:_ Compute concentration at the genre_l2 level (or on a relevance-filtered subset), not on the raw keyword cluster. Concretely: (1) keyword/synonym-filter games (require the query token in name/description) before computing demand/leader_share/HHI; (2) report a per-genre HHI/leader_share from genre_breakdown and make the VERDICT key off the dominant on-topic genre; (3) keep a small bleed-through denylist (the Candy & Chocolate keyboard game, etc.) or drop any game whose genre_l2 differs from the cluster's modal genre_l2 and whose name shares no token with the query, then recompute openness from the cleaned share.
- **[high] Demand component saturates far too early (demand_pts = min(35, 7*log10(demand+1)))** — The demand score reaches its 35.0 cap at demand ≈ 99,999 CCU, so it stops discriminating exactly in the range that matters. Evidence from score_breakdown: niche_food (166,929), niche_cooking (109,729), niche_restaurant (104,193), niche_rng (787,722) and niche_incremental (614,737) ALL receive demand=35.0 despite ~8x variation. A 110k-CCU niche and an 788k-CCU niche look identical on the dimension that should most strongly separate them. Worse, demand_ccu is summed over off-topic bleed-through, so it overstates on-topic addressable demand (the 167k 'food' total includes Squid Game X, My Movie, Hide or Die).
  - _Fix:_ Raise the cap / lower the multiplier so the curve still moves across 100k–1M (e.g. cap ~1M), and/or score demand on on-topic, genre-filtered CCU only. Split into 'total niche demand' vs 'demand in the most relevant genre_l2' and score the latter. Critically, compute demand from the relevance-filtered set so off-topic CCU does not inflate it.
- **[high] Freshness component saturates at 2 winners and ignores leader recency (freshness_pts = min(20, 10*len(fresh_winners)))** — Freshness maxes out at just 2 fresh (<=90d) 1k+ winners, so niche_cooking (fresh_winners=2) and niche_food (fresh_winners=7) both score 20.0 — it cannot tell a barely-minting niche from a hot one. It treats a fresh 1,100-CCU game and a fresh 90,000-CCU game identically, and counts fresh winners ANYWHERE in the contaminated cluster (a fresh horror game counts toward 'food' freshness). The 1k bar is also knife-edged (999 -> 0 pts, 1001 -> 10 pts). is_fresh keys on age_days (creation), so a dormant game that just relaunched is not 'fresh' even if it is the real momentum story.
  - _Fix:_ Make freshness continuous and CCU-weighted (e.g. sum of log(CCU) over on-topic fresh winners, scaled), raise the cap so 7 fresh outscores 2 fresh, and restrict to on-topic games. Use a soft winner band (ramp 500→1500 CCU) instead of a hard 1k cliff. Optionally add a relaunch/re-surge signal from days_since_update for older games.
- **[high] MONOLITH verdict gate uses contaminated cluster share -> false negatives and softened gates (analyze_niche verdict thresholds)** — The MONOLITH gate fires on leader_share>=0.6 OR hhi>=5000 computed on the cluster. Bleed-through dilutes the denominator, so a niche with a genuine per-genre monolith passes as 'OPEN & PROVEN.' Example: niche_food genre_breakdown shows 'Life' genre is 97.7%–99.9% owned by 'Work at a Pizza Place' and 'Childhood Game' is 99.9% 'Hide or Die!' — true monoliths at the genre level — yet the niche headline reads fragmented (hhi 585) and 'OPEN & PROVEN.' Conversely the 'incremental' cluster's 40.3% leader is a bleed-through keyboard game, so the gate correctly does NOT fire but for the wrong reason; the real on-topic incremental leaders ('Fish It!' 142k, 'Build A Ring Farm' 40k) are themselves large. The gate is driven by an artifact, not structure.
  - _Fix:_ Evaluate the MONOLITH / OPEN gates per dominant genre_l2 after relevance-filtering, and report the verdict for the specific lane the user would actually enter (e.g. 'cooking/restaurant Tycoon') rather than the keyword soup. Surface per-genre leader_share next to the verdict so closed sub-lanes (e.g. the 'Life'-genre pizza lane) are visible.
- **[medium] Openness double-counts concentration and can punish healthy 1k+ lanes (openness_pts = 35*(1 - max(leader_share, hhi/10000)))** — max(leader_share, hhi/10000) lets whichever contaminated proxy is larger dominate. In a healthy lane with one clear-but-non-dominant leader plus a deep tail, leader_share punishes openness even when a fresh 1k+ entrant is clearly feasible (the niche is still minting fresh winners). The component conflates 'concentrated' with 'unwinnable' and has no notion of how many independent 1k+ winners exist — the most direct evidence a new 1k+ entrant can succeed.
  - _Fix:_ Augment/replace openness with a 'reachability' signal: count of distinct on-topic 1k+ winners and the CCU gap between rank-3 and rank-10 (a deep mid-tier means room to slot in). Use HHI OR leader_share deliberately rather than max() of contaminated both, and cap the penalty so a niche provably minting fresh 1k+ entrants cannot score low on openness.
- **[medium] Score compression destroys ranking power across the user's candidate niches** — Because demand and freshness are saturated for all candidate niches and openness is the only mover, final scores cluster tightly: niche_food 85.9, niche_cooking 83.8, niche_restaurant 83.5, niche_rng 80.8, niche_incremental 75.9. A ~10-point spread across niches with ~8x demand difference and very different true concentration is not actionable — the user cannot rank lanes by this number, and the deep fields of independent winners (restaurant 21, cooking 19) get little credit.
  - _Fix:_ Recalibrate so each component stays informative across the realistic operating range (100k–1M demand; 1–10+ fresh winners; 5–25 total winners). Add de-contaminated 'winners_1k' as an explicit scored term. Validate by checking the four candidate lanes produce a meaningful spread that matches per-genre inspection.
- **[medium] False-positive / false-negative profile for the user's stated goal** — FALSE POSITIVES: (a) 'incremental' scores 75.9 / OPEN&PROVEN, but its on-topic Incremental-Simulator genre is led by a 247,976 bleed-through headline plus genuinely large incumbents Fish It! (142k) and Build A Ring Farm (40k) — a fresh RNG-collection entrant faces a crowded top end, so 'open' is partly an artifact. (b) niche_food reads ultra-fragmented (hhi 585) only because the cluster is padded with horror/grocery/movie games; the real food lane is thinner than 167k implies. FALSE NEGATIVES: the genre_breakdown reveals truly closed sub-lanes (food 'Life' 97.7%, 'Childhood Game' 99.9%) that the 'OPEN' headline hides — a user trusting the headline could walk into a single-owner lane. The method can also miss a strong open lane if its keyword pulls in a giant bleed-through game that trips the MONOLITH gate.
  - _Fix:_ Always present the per-genre breakdown beside the verdict and require the verdict to be confirmed at genre level. Flag any niche whose top games' modal genre_l2 differs from the user's target genre as 'contaminated — interpret with care.' For this user, anchor to genre_l2='Incremental Simulator' (Slime RNG lane) and 'Tycoon' (cooking/restaurant lane) and report concentration within those genres only.
- **[low] 1k winner bar and freshness rule are reasonable but knife-edged and snapshot-only** — WINNER_CCU=1000 is a defensible 'this lane can feed a 1k game' bar and matches the user's target, but it is a hard cliff (999 vs 1001) applied to a single live CCU snapshot with no peak/retention — a game spiking during an event reads identically to a steadily-retained one. visits_per_day is a lifetime average, not current growth, so it cannot corroborate momentum. The script header and report footer do correctly document the snapshot caveat.
  - _Fix:_ Soften the winner bar into a band (partial credit 500–1500) and, where feasible, cron the scanner to capture CCU over time so 'sustains 1k+' can be distinguished from 'spiked once.' Until history exists, label counts as 'snapshot 1k+ count' and avoid implying durability.
- **[low] Log-scaled demand shape and HHI bands are sound in principle** — Using log10 for demand is the right shape (diminishing returns on raw CCU), and HHI scaled 0–10000 with the >5000 / <1500 bands is a standard, defensible concentration framework. The problems are calibration (cap at ~100k) and contaminated input, not the use of logs or HHI per se.
  - _Fix:_ Keep the log shape and HHI bands; only move the demand cap and clean/relevance-filter the input set as described in the high-severity findings.
