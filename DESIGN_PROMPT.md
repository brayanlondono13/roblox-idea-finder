You are a senior product designer who specializes in **data-dense dashboards and tools for
creators/developers** (think Linear, Vercel Analytics, Steam Charts, Posthog). I want your
help designing the best possible version of a web dashboard. Below is everything about it.
Be opinionated, specific, and propose a complete design — not vague principles.

## What it is
**"Roblox Idea Finder"** — a single-page dashboard that helps a **solo / indie Roblox game
developer decide WHAT game to build next**, using live market data scraped from Roblox. The
core job-to-be-done: *"show me open, proven opportunities and tell me what to build right now —
and let me validate it in the data."*

Live site: https://brayanlondono13.github.io/roblox-idea-finder/

## Audience & goals
- **Primary user:** one indie Roblox dev (the owner). Secondary: other devs who find the public URL.
- **Wants:** an at-a-glance "what should I build right now," the ability to explore/validate an
  idea in the data, see what's trending, and avoid building a clone of an entrenched monolith.
- **Emotional goal:** feel like they have a quant's edge on a chaotic, trend-driven market.
- Roblox devs skew young and move fast; the market's metas burn hot and fade in weeks.

## Content the site presents (tabs today)
1. **Recommended Games (the hero).** Ranked "build this now" cards. Click a card → a full brief:
   core loop, first-30-seconds hook, depth (why it lasts), why-now evidence (with real CCU
   numbers), incumbent to beat, differentiation, monetization, virality, **biggest risk**,
   plus badges for category (proven/emerging/bold), confidence (high/med/low), and scope (S/M/L).
2. **"Right Now" strip (auto-refreshes daily):** open lanes (genres with no dominant leader),
   newest 1k+ CCU winners, hot mechanic×theme combos.
3. **Opportunity Map** — interactive scatter; each dot = a mechanic×theme/genre pair;
   x = reach (how popular the two ingredients are), y = how many games already combine them.
   "Opportunities" sit bottom-right (popular ingredients, rarely paired).
4. **Game Universe** — a t-SNE map of ~2,900 games clustered by similarity; color = genre,
   dot size = live concurrent players (CCU).
5. **Winning Ingredients** — bar chart of tags that over-index among 1,000+ CCU hits.
6. **Tables** — sortable "proven" and "untapped" combo lists.

## Current implementation & HARD constraints
- A **single self-contained `index.html`**, generated by a Python script that embeds all data
  as JSON. **No build step, no CDN, works offline by double-click.** Vanilla JS + `<canvas>`
  (no React, no D3, no charting lib). ~1 MB. Dark, GitHub-ish theme. Rebuilt daily.
- **Must stay:** one self-contained file; vanilla JS/canvas/SVG only; no external runtime deps;
  loads fast. **Can change freely:** layout, IA, color, typography, components, interactions, copy.
- Data values change daily — the design must handle **changing numbers and empty states**
  gracefully (e.g., a lane that closes, a card with thin evidence, 0 fresh winners).

## What I want from you (please cover all of these)
1. **Critique** the current design honestly — hierarchy, clarity, what's confusing or
   underwhelming, where attention goes vs. where it should.
2. **Information architecture & default view** — what should the user see first, and the
   narrative flow from "what to build" → "validate it" → "explore."
3. **Per-view design** — for EACH view above, the best way to present that data: chart type,
   layout, encodings, labels, interactions, what to emphasize vs. cut. Apply real data-viz
   judgment (e.g., is t-SNE the right call for the game map, or is there a more legible
   layout? are bars the best for "winning ingredients"? how to make the Opportunity Map's
   "bottom-right = good" instantly readable?).
4. **Visual system** — a semantic color palette (opportunity / risk / category, color-blind-safe),
   a typographic scale, spacing/density, and a reusable **card design** for the hero.
5. **The hero experience** — how a Recommended card should look, and the detail view: how to
   convey **confidence, risk, evidence, and scope at a glance** so the user trusts it fast.
6. **Responsive + accessibility** — mobile/tablet layout for canvas-heavy views; contrast,
   focus states, keyboard nav, and a categorical palette that survives color blindness.
7. **A prioritized, concrete change list (P0 / P1 / P2)** I can implement in vanilla JS, with
   **ASCII wireframes / mockups** for the 2–3 most important screens.

Be decisive. When you recommend a layout or chart, explain the reasoning and the tradeoff you're
making. Assume I can implement anything in vanilla HTML/CSS/JS/canvas/SVG. **End with the single
highest-impact change you'd ship first.**
